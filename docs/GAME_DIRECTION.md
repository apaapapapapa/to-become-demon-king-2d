# ゲーム方針

## この文書の役割

この文書は、ゲームとして何を目指すかを定義します。

実装構造やクラス責務は `TECHNICAL_DESIGN.md` と `ARCHITECTURE.md` に分離し、本書ではゲーム体験、ビジュアル、物語、コンテンツ開発の判断基準を扱います。

## 中核コンセプト

『To Become Demon King』は、手作り感のあるピクセルアートで表現するアイソメトリック2D／2.5D RPGを目指します。

プレイヤーは比較的弱い存在から始まり、探索、交流、戦闘、成長を重ねながら「魔王になる」ことを目指します。

「魔王になる」という言葉の意味は、単純な悪役化に限定しません。物語とゲームシステムの両方で、力、立場、責任、選択などを扱える余地を残します。

## 参考作品との距離

Witchbrookなどの作品は、アイソメトリック視点やピクセルアート表現の参考にしますが、直接的な模倣は行いません。

参考にするのは次のような表現上の方向性です。

- アイソメトリック視点の世界
- レイヤー感のあるピクセルアート環境
- 2Dアセットでありながら奥行きを感じられる構成
- 見分けやすいキャラクターシルエット
- 雰囲気を補強するライティング
- 探索とインタラクションを重視する画面設計

一方で、学園生活シミュレーションや大規模な生活AIなど、参考作品の機能規模をそのまま追従することは初期目標にしません。

## ゲームプレイの柱

ゲームの基本体験は次の4本柱で構成します。

1. 探索
2. キャラクターとのインタラクション
3. 戦闘
4. 成長／進行

新機能を追加する際は、これらのどの柱を強化するのかを明確にします。どの柱にも寄与しない大規模機能は、必要性が確認できるまで導入しません。

## 現在のプロトタイプ

現在のプロトタイプでは、次の基礎ループを確認できます。

```text
フィールドを移動する
  -> NPCとInteractする
  -> 訓練用の敵をAttackする
  -> HPを減らして倒す
  -> Dodgeで回避移動する
  -> Pauseでゲームプレイ入力とUI入力を切り替える
```

この段階の目的は、個々の機能を完成品品質にすることではなく、今後NPC、会話、敵AI、クエストなどを追加できる技術的な土台を確立することです。

## 次の縦切り目標

次に作る縦切りでは、単発の機能確認から「小さくても一連のゲーム体験が完結する状態」へ進めます。

想定する最小ループは次のとおりです。

```text
小さなエリアから開始
  -> NPCと会話する
  -> 簡単な目的を受け取る
  -> 探索または戦闘を行う
  -> 結果を持ち帰る
  -> 報酬または進行変化を受け取る
```

このループを成立させるため、今後はNPC会話、目的管理、敵AI、簡単な報酬／進行の順に追加します。

## 物語の開発方針

初期段階でシナリオ全体を固定しません。

先に定義するのは次の骨格です。

- 主人公が誰か
- 主人公がなぜ魔王を目指すのか
- 世界の雰囲気
- 中心となる対立
- プレイヤーの選択が何に影響するのか

実際のゲームシステムで表現できることを確認しながら、物語の詳細を段階的に具体化します。

## ビジュアル方針

### 視点

アイソメトリック、またはそれに近い固定視点を基本とします。

カメラ回転を前提にした3Dワールド設計にはせず、2Dアセットと描画順制御で奥行きを表現します。

### 環境

地形はIsometric Tilemapを基礎にし、表示要素を用途ごとに分離します。

- 地面
- 衝突
- 建物／構造物
- 小物
- 前景の遮蔽物
- エフェクト

主要な地形・建物・キャラクターはプロジェクト管理アセットへ移行し、実行時生成の単純図形はプロトタイプ専用の装飾や演出補助に限定します。

### キャラクター

現在のプレイヤー表現はプロトタイプ用です。

最終的なキャラクターアートは、次の条件を確認したうえで制作します。

- 画面上で必要なシルエットと視認性
- 移動速度と1タイルあたりの見た目の距離
- コリジョンサイズ
- 攻撃、回避、被弾など必要なアニメーション数
- NPCや敵とのサイズ比

### ライティング

URP 2D Lightingを利用可能な前提としますが、ゲームプレイが多数の高負荷ライトに依存する設計は避けます。

ライティングは雰囲気と視認性を補強するために使用し、性能目標と実機計測を踏まえて調整します。

## 対象プラットフォーム

第一ターゲット:

- Windows / Steam

将来の移植候補:

- Nintendoのコンソールプラットフォームなど

そのため、ゲームプレイの中核を次へ直接依存させません。

- マウス専用操作
- Steam固有API
- 任意の単一プラットフォームだけで成立する保存先やユーザー識別方式

プラットフォーム固有機能は、実際に必要になった段階で専用境界を追加します。

## コンテンツ開発の判断基準

今後の機能追加では、次の順序を基本とします。

1. 小さなプレイ可能ループを成立させる
2. プレイヤーが繰り返したくなるか確認する
3. 必要なデータ構造と保存対象を確定する
4. コンテンツ量を増やす
5. 規模に応じてロード、セーブ、Platform、最適化基盤を拡張する

技術基盤を先に無制限に作るのではなく、実際のゲーム体験が要求した境界を追加します。
