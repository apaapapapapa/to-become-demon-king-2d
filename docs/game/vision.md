# ゲームビジョン

## この文書の役割

この文書は、ゲームとして何を目指すかを定義します。

実装構造やクラス責務は `design/`、具体的な機能仕様は `specifications/` に分離し、本書ではゲーム体験、ビジュアル、物語、コンテンツ開発の判断基準を扱います。

## 中核コンセプト

『To Become Demon King』は、手作り感のあるピクセルアートで表現するアイソメトリック2D／2.5D RPGを目指します。

プレイヤーは比較的弱い存在から始まり、探索、交流、戦闘、成長を重ねながら「魔王になる」ことを目指します。

「魔王になる」という言葉の意味は単純な悪役化に限定せず、力、立場、責任、選択などを物語とゲームシステムの両方で扱える余地を残します。

## 参考作品との距離

Witchbrookなどの作品は、アイソメトリック視点やピクセルアート表現の参考にしますが、直接的な模倣は行いません。

参考にする方向性:

- アイソメトリック視点の世界
- レイヤー感のあるピクセルアート環境
- 2Dアセットでありながら奥行きを感じられる構成
- 見分けやすいキャラクターシルエット
- 雰囲気を補強するライティング
- 探索とインタラクションを重視する画面設計

学園生活シミュレーションや大規模な生活AIなど、参考作品の機能規模をそのまま追従することは目標にしません。

## ゲームプレイの柱

1. 探索
2. キャラクターとのインタラクション
3. 戦闘
4. 成長／進行

新機能を追加する際は、これらのどの柱を強化するのかを明確にします。

## 現在のプロトタイプ

```text
フィールドを移動する
  -> NPCとInteractする
  -> 訓練用の敵をAttackする
  -> HPを減らして倒す
  -> Dodgeで回避移動する
  -> PauseでGameplay入力とUI入力を切り替える
```

現在の目的は完成品品質のコンテンツ量ではなく、NPC、会話、敵AI、クエストなどを追加できる技術的な土台を確立することです。

## 次の縦切り目標

```text
小さなエリアから開始
  -> NPCと会話する
  -> 簡単な目的を受け取る
  -> 探索または戦闘を行う
  -> 結果を持ち帰る
  -> 報酬または進行変化を受け取る
```

このループを成立させるため、NPC会話、目的管理、敵AI、簡単な報酬／進行を段階的に追加します。

## 物語の開発方針

初期段階でシナリオ全体を固定しません。

先に定義する骨格:

- 主人公が誰か
- 主人公がなぜ魔王を目指すのか
- 世界の雰囲気
- 中心となる対立
- プレイヤーの選択が何に影響するのか

詳細は `story/` で管理します。

## ビジュアル方針

### 視点

アイソメトリック、またはそれに近い固定視点を基本とします。カメラ回転を前提にした3Dワールド設計にはせず、2Dアセットと描画順制御で奥行きを表現します。

### 環境

地形はIsometric Tilemapを基礎にし、地面、衝突、建物、小物、前景、エフェクトを用途ごとに分離します。

主要な地形・建物・キャラクターはプロジェクト管理アセットを正とし、実行時生成の単純図形はPrototype専用の補助表現に限定します。

### キャラクター

最終アートでは、シルエット、タイルスケール、コリジョン、攻撃・回避・被弾アニメーション、NPC・敵とのサイズ比を考慮します。

### ライティング

URP 2D Lightingを利用可能な前提としますが、多数の高負荷ライトへゲームプレイを依存させません。雰囲気と視認性を補強するために使用します。

## 対象プラットフォーム

第一ターゲットはWindows / Steamです。将来のコンソール移植を妨げないよう、ゲームプレイの中核をSteam固有APIやマウス専用操作へ直接依存させません。

## コンテンツ開発の判断基準

1. 小さなプレイ可能ループを成立させる
2. プレイヤーが繰り返したくなるか確認する
3. 必要なデータ構造と保存対象を確定する
4. コンテンツ量を増やす
5. 規模に応じてロード、セーブ、Platform、最適化基盤を拡張する
